## 1. PRINCIPIOS FUNDAMENTALES

### 1.1 Principios de Desarrollo

| Principio | Descripción | Aplicación |
| :--- | :--- | :--- |
| **Incrementalidad** | Desarrollamos funcionalidad por funcionalidad, nunca todo a la vez. Cada entrega es un módulo completo y funcional. | Una clase o grupo de clases relacionadas por entrega |
| **Validación Continua** | Cada módulo se valida antes de pasar al siguiente. No acumulamos deuda técnica. | Revisión de RFs, compilación, verificación de dependencias |
| **Documentación Integrada** | Cada archivo .mqh incluye su propia documentación (encabezado, descripción de métodos, ejemplos). | Documentación inline, comentarios de método |
| **Trazabilidad** | Cada línea de código debe ser trazable a un RF específico del SRS. | Comentarios con RFs al inicio de cada método |
| **Optimización por Diseño** | La eficiencia se diseña desde el principio, no se parchea después. | Caching, batching, timer-based execution en el diseño |
| **Bajo Acoplamiento** | Las clases se comunican a través de interfaces bien definidas, minimizando dependencias directas. | Uso de referencias, no herencia innecesaria |
| **Alta Cohesión** | Cada clase tiene una única responsabilidad (Single Responsibility Principle). | Una clase = una responsabilidad principal |

### 1.2 Principios de Comunicación

| Principio | Descripción |
| :--- | :--- |
| **Entregas Claras** | Cada entrega incluye: código fuente, RFs cubiertos, dependencias, instrucciones de integración, próximos pasos |
| **Actualización Constante** | El estado del proyecto se actualiza después de cada entrega |
| **Preguntas Abiertas** | Se formulan preguntas claras cuando hay ambigüedad |
| **Validación Cruzada** | Se valida contra SRS, Arquitectura y Contexto |

### 2.2 Descripción Detallada de Cada Paso

#### Paso 1: Revisión de Requisitos

**Objetivo:** Entender exactamente qué se debe implementar.

**Acciones:**
1. Leer los RFs correspondientes en el SRS v1.2.
2. Identificar las estructuras de datos necesarias.
3. Verificar dependencias (¿qué otras clases necesita esta clase?).
4. Identificar posibles ambigüedades y resolverlas.

**Entregable:** Lista de RFs a implementar y dependencias identificadas.

#### Paso 2: Diseño de Clase/Módulo

**Objetivo:** Definir la estructura de la clase antes de escribir código.

**Acciones:**
1. Definir miembros privados (variables de estado, referencias a otras clases).
2. Definir métodos públicos (interfaz que usará CHunterIPDA y otras clases).
3. Definir métodos privados (lógica interna).
4. Documentar cada método con su propósito, parámetros y retorno.

**Entregable:** Esquema de la clase con todos sus métodos definidos.

#### Paso 3: Verificación con Documentación Oficial MQL5

**Objetivo:** Confirmar que la implementación es correcta según MQL5.

**Fuentes de verificación obligatorias:**

| Fuente | Descripción |
| :--- | :--- |
| **Documentación oficial MQL5** | https://www.mql5.com/en/docs |
| **Reglas críticas de sintaxis** | No usar `->` para acceder a miembros; usar `.` |
| **Estructura de archivos** | Usar `#ifndef`, `#define`, `#endif` como guards |
| **Orden de includes** | Sistema → Librería → DataTypes → Utils → Componentes |
| **Manejo de errores** | Verificar `GetLastError()` y retornos de funciones |

**Reglas críticas de MQL5 a respetar:**

| Regla | Descripción | Ejemplo |
| :--- | :--- | :--- |
| **NO usar `->`** | Nunca usar puntero sintaxis `->`; siempre usar `.` | `m_utils.Log()` en lugar de `m_utils->Log()` |
| **Referencias solo para arrays** | No usar referencias para variables locales simples | `void Function(int &arr[])` OK; `void Function(int &value)` NO |
| **Enums antes de estructuras** | Definir enums ANTES de las estructuras que los usan | `enum ENUM_EXIT_REASON` antes de `struct ManagedPosition` |
| **Objetos: declarar con `*`, acceder con `.`** | Los punteros se declaran con `*` pero se accede con `.` | `CConfig* m_config; m_config.Init();` |
| **Verificar `NULL` antes de usar** | Siempre validar punteros antes de usarlos | `if(m_utils != NULL) m_utils.Log(...)` |

#### Paso 4: Preguntar Antes de Implementar

**Objetivo:** Asegurar que el usuario está alineado con la implementación.

**Preguntas obligatorias:**
1. ¿Desea crear el archivo `[Nombre].mqh`?
2. ¿Quiere ver el resumen de la entrega o el código completo?
3. ¿Alguna preferencia específica para la estructura?

#### Paso 5: Implementación del Código

**Objetivo:** Escribir el código fuente siguiendo el estándar.

**Acciones:**
1. Escribir el código método por método.
2. Asegurar que cada método tenga manejo de errores.
3. Incluir logging para trazabilidad.
4. Mantener el código limpio y bien estructurado.
5. Añadir entrada en CHANGELOG.

**Entregable:** Archivo .mqh completo y compilable.

#### Paso 6: Revisión y Validación

**Objetivo:** Verificar que el código funciona y cumple con los requisitos.

**Acciones:**
1. Compilar en MQL5 (verificar que no hay errores ni advertencias).
2. Verificar cobertura de RFs (todos los RFs de la fase están implementados).
3. Verificar dependencias (todas las dependencias están resueltas).
4. Verificar manejo de errores (todas las condiciones de error están cubiertas).

**Entregable:** Informe de validación.

#### Paso 7: Entrega y Actualización

**Objetivo:** Entregar el código y actualizar la documentación.

**Acciones:**
1. Entregar el código fuente (solo si se solicita).
2. Listar los RFs cubiertos.
3. Listar las dependencias.
4. Proporcionar instrucciones de integración.
5. Sugerir próximos pasos.
6. Actualizar el Contexto de la Conversación.

**Entregable:** Código entregado, documentación actualizada.

### 3.2 Reglas de Nomenclatura

| Elemento | Regla | Ejemplo |
| :--- | :--- | :--- |
| **Clases** | Prefijo `C` + PascalCase | `CConfig`, `CDetector`, `CBonusHunter` |
| **Métodos** | PascalCase | `CalculateLotSize()`, `DetectFVG()` |
| **Variables miembro** | Prefijo `m_` + camelCase | `m_config`, `m_isInitialized` |
| **Variables locales** | camelCase | `lotSize`, `stopDistance` |
| **Constantes** | MAYÚSCULAS + guión bajo | `MAX_SPREAD_SCALPING`, `RISK_POSITION_DEFAULT` |
| **Enumeraciones** | Prefijo `ENUM_` + MAYÚSCULAS | `ENUM_BIAS`, `ENUM_EA_STATE` |
| **Estructuras** | PascalCase | `OrderBlock`, `Signal`, `MegaTradeData` |
| **Archivos** | Misma que la clase | `CConfig.mqh`, `CDetector.mqh` |

### 3.5 Manejo de Errores

#### Patrón Estándar

```cpp
//--- Verificación de parámetros
if(param == NULL) {
    Print(ClassName + "::MethodName - Error: Parámetro NULL");
    return false;
}

//--- Verificación de inicialización
if(!m_isInitialized) {
    Print(ClassName + "::MethodName - Error: Módulo no inicializado");
    return false;
}

//--- Verificación de condiciones de mercado
if(!m_utils.IsMarketOpen(symbol)) {
    Print(ClassName + "::MethodName - Error: Mercado cerrado");
    return false;
}

//--- Verificación de resultados
if(result <= 0) {
    Print(ClassName + "::MethodName - Error: Resultado inválido: " + DoubleToString(result));
    return false;
}
```

#### Función de Logging

```cpp
//--- Logging centralizado
void LogError(string message, int errorCode = 0) {
    string errorMsg = "ERROR: " + message;
    if(errorCode > 0) {
        errorMsg += " (código: " + IntegerToString(errorCode) + ")";
    }
    Print(errorMsg);
    
    //--- Si CLogger está disponible
    if(m_logger != NULL) {
        m_logger.LogError(message, errorCode);
    }
}

void LogWarning(string message) {
    Print("WARNING: " + message);
    if(m_logger != NULL) {
        m_logger.LogWarning(message);
    }
}

void LogInfo(string message) {
    Print("INFO: " + message);
    if(m_logger != NULL) {
        m_logger.LogInfo(message);
    }
}
```

## 4. REGLAS DE ENTREGA

### 4.1 Lista de Verificación de Entrega

**Antes de cada entrega, verificar:**

- [ ] **Compilación:** El código compila sin errores ni advertencias en MQL5.
- [ ] **RFs Cubiertos:** Todos los RFs de la fase están implementados y documentados.
- [ ] **Dependencias:** Se han resuelto todas las dependencias con módulos anteriores.
- [ ] **Documentación:** Cada método tiene su documentación completa.
- [ ] **Manejo de Errores:** Todas las condiciones de error están manejadas.
- [ ] **Logging:** El módulo registra sus acciones importantes (entradas, salidas, errores).
- [ ] **Estructura:** El código sigue el estándar de codificación definido.
- [ ] **Trazabilidad:** Cada RF tiene su implementación identificable.
- [ ] **Changelog:** El archivo tiene una entrada en CHANGELOG.

### 4.2 Reglas de Entrega de Archivos

| Regla | Descripción |
| :--- | :--- |
| **Preguntar Antes de Crear** | Antes de crear cualquier archivo nuevo, el desarrollador debe preguntar explícitamente si el usuario desea que se cree. |
| **Archivos Nuevos en .txt** | Cuando se crea un archivo nuevo, se debe entregar en formato .txt (no .mqh) para evitar que el sistema lo interprete como código ejecutable. |
| **Modificaciones en Diff** | Cuando se modifica un archivo existente, se debe entregar el código en formato diff con los signos + (verde) y - (rojo). |
| **No Mostrar Archivos sin Solicitud** | El desarrollador no mostrará el contenido de archivos completos a menos que el usuario lo solicite específicamente. |
| **Formato Diff para Modificaciones** | Para modificaciones, se usará el formato diff de Git con signos + (verde) y - (rojo) en TODA la línea. |
| **Referencia de Líneas** | Cada modificación debe incluir el número de línea donde comienza el cambio. |

### 4.3 Formato Diff para Modificaciones

**REGLA:** El signo se aplica a TODA la línea, no solo a una parte.

**Ejemplo correcto:**

```diff
  //--- Líneas sin cambios (blanco)
+ //--- TODA esta línea es verde (línea añadida)
+ //--- TODA esta línea también es verde
- //--- TODA esta línea es roja (línea eliminada)
- //--- TODA esta línea también es roja
  //--- Líneas sin cambios (blanco)
```

**Estructura de la entrega:**

```
📁 **Ubicación:** `Include/[Dominio]/[Nombre].mqh`
📄 **Estado:** MODIFICADO
🔢 **Línea de inicio:** [Número de línea donde comienza el cambio]
📝 **Cambio:** [Breve descripción]

**Código MODIFICADO:**
```diff
  //--- Líneas sin cambios (blanco)
+ //--- Línea añadida (verde)
+ //--- Otra línea añadida (verde)
- //--- Línea eliminada (rojo)
- //--- Otra línea eliminada (rojo)
  //--- Líneas sin cambios (blanco)
```
```

### 4.4 Formato de Entrega de Archivos Nuevos

**Para ARCHIVOS NUEVOS (primera implementación):**

```
📁 **Ubicación:** `Include/[Dominio]/[Nombre].mqh`
📄 **Estado:** NUEVO ARCHIVO (PRE-APROBACIÓN)
📝 **Descripción:** [Breve descripción del módulo]
⚠️ **Pendiente de aprobación para crear el archivo**
```

   if(m_[nombre] != NULL) { delete m_[nombre]; m_[nombre] = NULL; }

### 7.2 Gestión de Dependencias Circulares

**Regla:** No deben existir dependencias circulares.

**Solución:** Si se detecta una dependencia circular, se debe:
1. Evaluar si la dependencia es realmente necesaria.
2. Si es necesaria, extraer la funcionalidad compartida en una clase separada.
3. Si no es necesaria, eliminar la dependencia.

## 8. MANEJO DE ERRORES Y LOGGING

### 8.1 Jerarquía de Logging

| Nivel | Descripción | Uso |
| :--- | :--- | :--- |
| **ERROR** | Errores que impiden el funcionamiento | Fallos de inicialización, errores de ejecución de órdenes, errores de validación |
| **WARNING** | Advertencias que no impiden el funcionamiento | Parámetros fuera de rango, condiciones de mercado no ideales |
| **INFO** | Información importante | Inicialización de módulos, apertura/cierre de operaciones, cambios de estado |
| **DEBUG** | Información de depuración | Detalle de cálculos, valores de variables, estado interno |
| **TRACE** | Traza completa | Todas las acciones, incluyendo las más detalladas |

### 8.2 Estructura de Mensajes de Log

```
[YYYY-MM-DD HH:MM:SS] [NIVEL] [MÓDULO] Mensaje
```

**Ejemplo:**
```
[2026-07-21 14:30:25] [INFO] [CExecutor] Orden ejecutada: BUY 0.1 EURUSD @ 1.12345, SL: 1.12245, TP: 1.12545
```
